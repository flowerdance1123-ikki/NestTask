# CLAUDE.md

このファイルは、NestTask プロジェクトで AI コーディングエージェントが作業するときの前提・設計方針・実装ルールをまとめたものです。

NestTask は「普通の Todo アプリ」ではありません。
アプリの中核コンセプトを崩さないように実装してください。

---

## 1. アプリ概要

### アプリ名

NestTask

### 一言コンセプト

NestTask は、毎月・毎週くり返す複雑な業務や、必要な時に何度も使う作業手順を、親タスクと子タスクのテンプレートとして管理し、実行時にチェックで潰していける階層式タスク管理アプリです。

### 重要な考え方

普通の Todo アプリは、タスクが「一期一会」になりがちです。

```text
思いつく
↓
書く
↓
終わったら消える
```

NestTask は違います。

```text
業務の型を作る
↓
予定日または必要な時に呼び出す
↓
親子タスクとして展開する
↓
チェックで潰す
↓
履歴が残る
↓
次回も同じ型を使える
```

NestTask の価値は、タスクを使い捨てにせず、何度も使える「作業テンプレート」として扱うことです。

---

## 2. NestTask が解決する課題

ユーザーは、以下のような作業で困っています。

* 毎月発生することは分かっているが、細かい作業が多い
* 毎回「何をやるんだっけ？」と思い出す必要がある
* 普通の Todo アプリでは、親タスクの中身が見えず、頭の中が散らかる
* Notion や Obsidian ほど重い管理はしたくない
* マインドマップ的に作業を分解したいが、実行時はチェックリストで潰したい
* 業務手順を一度作ったら、次回以降も使い回したい

NestTask は、こうした「くり返し発生する複雑な業務」を階層化し、テンプレート化し、実行可能なチェックリストに変換するアプリです。

---

## 3. 想定ユーザー

主な想定ユーザーは、単発タスクではなく「まとまった業務」を抱えている人です。

例：

* 月末業務がある人
* 週次・月次の定型業務が多い人
* 個人事業主
* 小規模事業者
* 現場職と事務作業を兼任している人
* 医療・介護・訪問系の業務担当者
* バックオフィス業務を兼任している人
* 毎回同じような準備作業をする人
* チェックリスト運用をしたい人

ただし、特定業界専用アプリにはしません。
汎用的な「くり返し業務テンプレート実行アプリ」として設計します。

## 4. コア機能

NestTask の中核機能は以下です。

### 4.1 テンプレート

何度も使う作業の型です。

例：

```text
月末請求業務テンプレート
├─ 請求データ確認
│  ├─ SpotLogを開く
│  ├─ 対象患者を確認
│  └─ 保険・自費を分ける
├─ PDF作成
│  ├─ 請求書PDFを作成
│  ├─ 領収書PDFを作成
│  └─ ファイル名を確認
└─ 提出前チェック
   ├─ 印刷内容を確認
   ├─ 提出先を確認
   └─ 完了報告する
```

テンプレートは直接チェックして完了する対象ではありません。
テンプレートは「型」です。

### 4.2 実行タスク

テンプレートから作成された、今回分のタスクです。

例：

```text
2026年6月 月末請求業務
├─ 請求データ確認
│  ├─ SpotLogを開く
│  ├─ 対象患者を確認
│  └─ 保険・自費を分ける
├─ PDF作成
│  ├─ 請求書PDFを作成
│  ├─ 領収書PDFを作成
│  └─ ファイル名を確認
└─ 提出前チェック
   ├─ 印刷内容を確認
   ├─ 提出先を確認
   └─ 完了報告する
```

ユーザーがチェックして潰すのは、テンプレートではなく実行タスクです。

### 4.3 予定ありテンプレート

毎日・毎週・毎月・毎年など、予定日に自動で実行タスクとして出てくるテンプレートです。

例：

* 毎月25日：月末請求業務
* 毎週金曜：AIアシスト通信作成
* 毎月1日：経費精算チェック
* 毎年4月：年度更新作業

### 4.4 予定なしテンプレート

必要な時に手動で呼び出すテンプレートです。

例：

* 新規患者対応
* セミナー準備
* 外販ヒアリング
* アプリ導入手順
* 問い合わせ対応
* 旅行準備

予定が決まっていなくても、作業手順として保存しておき、必要な時に「テンプレートから開始」できます。

### 4.5 履歴

実行したタスクは履歴に残ります。

履歴には以下を残します。

* 実行タスク名
* 元になったテンプレート
* 開始日
* 期限
* 完了日
* 完了率
* 未完了項目
* 実行時メモ

履歴は「前回どうやったか」を確認するために重要です。

### 4.6 テンプレート改善

実行中に項目を追加した場合、以下の選択肢を出せる設計にします。

```text
この項目をテンプレートにも追加しますか？
- テンプレートにも追加
- 今回だけ追加
```

これにより、使うほどテンプレートが育ちます。

## 5. 画面構成

現在の基本タブは以下の4つです。

### 5.1 今日

今日やるべき実行タスクを表示するメイン画面です。

表示対象：

* 今日開始の実行タスク
* 今日が期限の実行タスク
* 期限切れの実行タスク
* 手動で開始した実行タスク
* 表示開始日を迎えた予定タスク

主な機能：

* 親タスクカード表示
* 進捗表示
* 展開・折りたたみ
* 子タスクのチェック
* 完了状態の更新

### 5.2 予定

予定ありテンプレートがいつ発生するか確認する画面です。

最初はアプリ内カレンダーとして実装します。
Appleカレンダーとの同期は初期版では不要です。

表示例：

* 月間カレンダー
* 日付ごとの予定ドット
* 選択日の予定タスク
* 今後7日間の予定

### 5.3 テンプレート

テンプレートを管理する画面です。

表示切り替え：

* すべて
* 予定あり
* 予定なし
* お気に入り
* カテゴリ別

主な機能：

* テンプレート作成
* テンプレート編集
* テンプレート削除
* テンプレート複製
* テンプレートから開始
* 予定設定

### 5.4 履歴

過去に実行したタスクを確認する画面です。

表示対象：

* 完了済み実行タスク
* 一部未完了の実行タスク
* 手動開始した実行タスク
* 予定から発生した実行タスク

## 6. データ設計方針

重要：テンプレートと実行タスクは必ず分けて扱ってください。

テンプレートを直接チェックしてはいけません。
チェック状態を持つのは実行タスク側です。

### 6.1 推奨モデル構成

初期実装では SwiftData を想定します。

#### TaskTemplate

テンプレート本体です。

想定プロパティ：

```text
id: UUID
title: String
memo: String?
category: String?
colorHex: String?
iconName: String?
isFavorite: Bool
isArchived: Bool
scheduleType: ScheduleType
createdAt: Date
updatedAt: Date
```

#### TemplateNode

テンプレート内の親子タスク構造です。

想定プロパティ：

```text
id: UUID
templateId: UUID
parentId: UUID?
title: String
memo: String?
orderIndex: Int
createdAt: Date
updatedAt: Date
```

#### TaskRun

テンプレートから生成された今回分の実行タスクです。

想定プロパティ：

```text
id: UUID
templateId: UUID?
title: String
startDate: Date
dueDate: Date?
status: TaskRunStatus
createdAt: Date
updatedAt: Date
completedAt: Date?
```

#### TaskNode

実行タスク内の親子タスク構造です。
ユーザーがチェックするのはこの TaskNode です。

想定プロパティ：

```text
id: UUID
taskRunId: UUID
templateNodeId: UUID?
parentId: UUID?
title: String
memo: String?
orderIndex: Int
isCompleted: Bool
createdAt: Date
updatedAt: Date
completedAt: Date?
```

## 7. スケジュール設計

テンプレートには発生方法を持たせます。

### 7.1 初期版で扱う発生方法

* 予定なし：必要な時に開始
* 毎日
* 毎週：曜日指定
* 毎月：日付指定
* 毎月末
* 毎年：月日指定

### 7.2 事前表示

予定日にいきなり表示されるだけでは遅い場合があるため、表示開始設定を持たせます。

例：

```text
発生日：毎月25日
表示開始：3日前
期限：月末
通知：前日9:00
```

初期版では、以下程度で十分です。

* 当日に表示
* 1日前に表示
* 3日前に表示
* 7日前に表示

### 7.3 実行タスク生成ルール

予定ありテンプレートは、表示開始日または発生日を迎えたら実行タスクとして生成されます。

重要：

* 同じテンプレート・同じ対象日で重複生成しない
* アプリ起動時に「今日までに生成されるべき実行タスク」を確認する
* iOS のバックグラウンド処理に過度に依存しない
* ローカル通知はアプリを開くきっかけとして使う
* 生成処理は idempotent、つまり何度実行しても重複しない形にする

## 8. ローカル保存方針

初期版はローカル保存を基本とします。

入れないもの：

* アカウント登録
* サーバー同期
* チーム共有
* 外部バックエンド
* AI連携
* 外部カレンダー双方向同期

ローカル保存にする理由：

* 開発範囲を絞れる
* App Store 審査のリスクを下げられる
* プライバシー説明が簡単
* 業務情報や個人情報を入力しやすい
* オフラインでも使える

将来的には iCloud 同期やバックアップを検討しますが、初期実装では必須ではありません。

---

## 9. バックアップ・エクスポート方針

ローカル保存だけだと端末変更やアプリ削除時にデータ消失リスクがあります。

初期版では最低限、以下を検討します。

* 全データの JSON エクスポート
* 全データの JSON インポート
* テンプレートのみエクスポート
* テンプレートのみインポート

ただし、最初のバックエンド実装では SwiftData の基本保存を優先します。
エクスポート・インポートは後続タスクで構いません。

## 11. タスク階層の考え方

階層は自由度を持たせますが、UIとしては深くしすぎない方針です。

基本の考え方：

```text
親タスク：業務のまとまり
子タスク：作業工程
孫タスク：具体的なチェック項目
```

初期 UI では 3 階層程度が快適に見える設計を優先します。

内部的には parentId によるツリー構造を採用します。
ただし、画面表示時は必要に応じてフラット化して扱って構いません。

## 12. 完了・進捗ルール

### 12.1 子タスク

子タスクをチェックすると完了扱いになります。

完了済み子タスクは以下の表示にします。

* チェック済みアイコン
* 薄い文字色
* 必要に応じて取り消し線
* 復元可能

### 12.2 親タスク

親タスクの進捗は、配下の子タスクの完了率から計算します。

例：

```text
12項目中7項目完了
58%
```

子タスクがすべて完了したら、親タスクも自動完了扱いにして構いません。

ただし、親タスクを直接チェックする場合は、配下の子タスクをすべて完了にするか確認する UI を検討してください。

### 12.3 実行タスク全体

実行タスク全体の進捗は、配下の TaskNode の完了率から計算します。

TaskRun の status は以下のような値を想定します。

```swift
enum TaskRunStatus {
    case active
    case completed
    case archived
}
```

## 15. 実装時の禁止事項・注意事項

### 15.1 普通の Todo アプリにしない

NestTask は単発 Todo を大量に並べるアプリではありません。

必ず以下を中心に設計してください。

* テンプレート
* 実行タスク
* 親子タスク
* 予定日または手動開始
* 履歴
* テンプレート改善

### 15.2 テンプレートにチェック状態を持たせない

テンプレートは型です。
チェック状態を持つのは実行タスク側です。

### 15.3 予定機能を外部カレンダー前提にしない

初期版ではアプリ内予定で十分です。
Apple カレンダー連携や双方向同期は後回しです。

### 15.4 クラウド同期を勝手に入れない

初期版はローカル保存です。
サーバー、アカウント登録、外部同期は入れないでください。

### 15.5 AI 機能を勝手に入れない

AI によるタスク自動分解は将来的な拡張候補ですが、初期版では不要です。

### 15.6 過度に複雑なプロジェクト管理にしない

ガントチャート、チーム管理、コメント、承認フローなどは初期版では不要です。
NestTask は軽く使える「定型業務テンプレート実行アプリ」です。

## 18. バックエンド実装の最初のゴール

最初のゴールは、以下の流れが実データで動くことです。

```text
テンプレートを1つ作る
↓
親タスク・子タスクを登録する
↓
テンプレートから実行タスクを生成する
↓
今日画面に実行タスクが表示される
↓
子タスクをチェックする
↓
進捗が保存される
↓
アプリを閉じて開いても状態が残る
```

まずはこの流れを完成させてください。

予定生成、通知、履歴、バックアップはその後で構いません。

## 19. 現在の重要判断

このプロジェクトでは、以下の判断を採用します。

* 初期版は iOS アプリ
* SwiftUI で実装
* SwiftData によるローカル保存を基本とする
* アカウント登録なし
* クラウド同期なし
* 外部バックエンドなし
* テンプレートと実行タスクを分離する
* 予定ありテンプレートと予定なしテンプレートを両方扱う
* 今日 / 予定 / テンプレート / 履歴 の4タブ構成
* App Store リリースを目指す
* まずは完成度の高いローカルファーストアプリを目指す
